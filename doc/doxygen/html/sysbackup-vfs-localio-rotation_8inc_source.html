00003  *
00004  * Local IO Rotation Plugin
00005  *
00006  * A plugin to automatically rotate backupsets so that a certain
00007  * number are online at all times (without going over that number.)
00008  * This is intended to help automatically manage disk space.
00009  * Configuration information inherited by this class is not actually
00010  * used - this class refers to the config info in the associated
00011  * localio VFS object.
00012  *
00013  * Note that these rotation plugins are not actually loaded as plugins
00014  * or as virtual filesystems. These are objects that interact with the
00015  * corresponding VFS objects. They are implemented as separate VFS
00016  * plugins so that they can load hash tables for the rotation folders
00017  * using the same code that is used by the VFS plugins.
00018  *
00019  * <p><b>What's a Rotation anyway?</b>
00020  *
00021  * Many years ago backups were written to tapes - large reels of
00022  * tape. Tape was expensive and difficult to work with. It would
00023  * become unreadable if it got crinkled and there were many little
00024  * problems that would cause the tape to go flying off the reel and
00025  * end-up all over the floor in a big mess. In my personal case I only
00026  * used reels of tape for music in the 70's. By the 80's tape had
00027  * fallen out of favor among audiophiles and tape cassettes had become
00028  * the more common format. Data, however, was still being recorded on
00029  * large tape reels.
00030  *
00031  * Personally I did not start using tape for data backup until some
00032  * time in the 80's when data cartridges became popular - so I never
00033  * really had to suffer the horror of big reels of data tape all over
00034  * the office floor. However I did have my share of problems with data
00035  * cartridges. They were a huge improvement over reels of tape. They
00036  * were easier to use and the tape rarely came out of the cartridge
00037  * where it belonged. Still, cartridges were expensive and, again, the
00038  * tape would easily become unreadable after being used a few times.
00039  *
00040  * So, we needed to be able to record our nightly backups onto the
00041  * cartridges without having the tape get crinkled and become
00042  * unreadable. Unfortunately the cartridges were painfully
00043  * expensive. We didn't have enough money to go out and buy one
00044  * cartridge for each day of the year. The most effective course of
00045  * action was to buy as many cartridges as we could afford and use
00046  * them sequentially. When we got to the end of a stack of cartridges
00047  * we would go back to the first one and start again. This was called
00048  * a ROTATION and it allowed all the previously used tapes in the
00049  * stack to be available in case of a problem. 
00050  *
00051  * In practice the tape would get crinkled and the cartridges would be
00052  * found unreadable after a few uses. If the previously recorded
00053  * backup had become unreadable due to crinkles in the tape... we just
00054  * tried the one before that, and so on, until we found one that was
00055  * readable. Sometimes we would manage to recover some files from one
00056  * cartridge and would look for more files on other cartridges. This
00057  * was not fun.
00058  *
00059  * We also used additional weekly and monthly rotations. This was in
00060  * part to extend the duration of the backup so that backupsets
00061  * covered a longer period of time. However, it was also a way of
00062  * ensuring that the cartridges would last longer before becoming
00063  * unreadable.
00064  *
00065  * The rotation process for the weekly and monthly backups was as
00066  * simple as the daily rotation process: Every Sunday the tape from
00067  * the previous Sunday would be rotated into a weekly backup
00068  * stack. Every month we might (if we were able to purchase enough
00069  * cartridges,) also rotate the weekly backup from the start of the previous
00070  * month into a monthly backup stack. Each week the expired weekly and
00071  * monthly cartridges would then go back into the daily rotation. This
00072  * ensured that there were always tapes in the daily rotation that
00073  * were less heavily used and, therefore, more likely to be reliable.
00074  *
00075  * As you can imagine this was a frustrating exercise.  Consequently
00076  * tape cartridges were never really very popular. Over time the cost
00077  * of hard disk drives had gone down to the point where some companies
00078  * starting using hard disk storage instead of tape storage in their
00079  * backup rotations. I had one customer in the 90s who had a computer
00080  * dedicated to the task of making backups. The drive bays on that
00081  * computer were all removable. The president of the company would
00082  * take one weekly backup home on Friday night and, on Monday morning,
00083  * would return with the previous weekly backup drive and put it back
00084  * into the company hard disk rotation. This ensured that the company
00085  * would have an off-site backup available (in his home) in the event
00086  * of a disaster at the office.
00087  *
00088  * In the current age of LVM snapshots, inexpensive hard disk storage,
00089  * Network Addressable storage and Cloud-based storage there are a
00090  * wide range of backup options available. It is now possible to take
00091  * a snapshot of a busy filesystem, back it up to a long-term local
00092  * rotation and copy it simultaneously to remote storage - possibly
00093  * even off-site.
00094  *
00095  * The availability of a long-term rotation on a local server improves
00096  * dramatically the response to typical user issues (especially if the
00097  * backup is fine-grained.) A user who has accidentally deleted a
00098  * file, for example, might not realize for several days that he has a
00099  * problem. When he does finally call the I.T. department for help the
00100  * daily rotation may have expired - but a copy of the file might be
00101  * found on the weekly and/or monthly rotations. By keeping the
00102  * backups as fine-grained as possible it is easy to find an
00103  * appropriate tarball and make it available to a user whenever
00104  * necessary.
00105  *
00106  * <p><b>How it works</b>
00107  *
00108  * <p>Files are added to this Virtual File System as they are for the
00109  * localio Virtual File System. However, the LocalIO VFS must be
00110  * allowed to work as quickly as possible to ensure that filesystem
00111  * snapshots are released quickly. Therefore the rotations plugins are
00112  * designed to queue any files that need to be copied and copy them
00113  * after the main backupset is created.
00114  *
00115  * <p>Specifically:
00116  *
00117  * <ul>
00118  *
00119  * <li>On load this plugin determines if it must write weekly or
00120  * monthly backups and sets appropriate flags.
00121  *
00122  * <li>On startup this plugin follows the same procedure that was
00123  * followed by the localio plugin except that it operates on the
00124  * rotation folder instead of the backup folder. In other words it
00125  * finds the previously written backupset in the Rotation folder (if
00126  * any) and loads the associated hashlist.
00127  *
00128  * <li>If weekly and/or monthly rotations are being written: All files
00129  * written to the main backupset are copied to the weekly and/or
00130  * monthly rotations. Hardlinks are always used if the same partitions
00131  * are used by the backup and rotations folders. If the partitions are
00132  * different the current hashlist is used to determine if an exact
00133  * copy of a file already exists in the previously written backupset
00134  * on the rotations folder. If so a hardlink is used; if not the file
00135  * is queued for later copying.
00136  *
00137  * <li>When all other tasks are completed the following tasks will be
00138  * executed:
00139  *
00140  * <p>(1) All queued file copy operations will be completed.
00141  *
00142  * <p>(2) The daily, weekly and monthly folders will be pruned. See
00143  * the notes below. Pruning is not done before the backup in case the
00144  * process fails. This ensures that there is at least one good backup
00145  * available on the local filesystem (the previous one, if any.)
00146  * Unfortunately this requires that local filesystems, if they will be
00147  * keeping any local copies of backups at all, require enough disk
00148  * space available at all times for at least two complete backups. (If
00149  * this is a problem on some VM's the last backup can always be
00150  * removed before creating the next backup.)
00151  *
00152  * <p><b>Pruning Folders</b>
00153  *
00154  * <p>This plugin takes a simple approach to managing rotations:
00155  *
00156  * <ul>
00157  *
00158  * <li>Daily rotations are pruned every day. The total number of
00159  * folders allowed in the Daily rotation folder (the folder where the
00160  * backupsets are written each day,) is limited to the number of
00161  * folders specified in the $this->app->localsys->rotation_total_daily
00162  * configuration parameter. The folders are sorted by creation date
00163  * and all but the most recent total_daily folders are deleted.
00164  *
00165  * <li>On one day of the week (specified in the
00166  * $this->app->localsys->rotation_dayofweek configuration parameter,)
00167  * the backupset that was just created is copied to the weekly backup
00168  * rotation. If the weekly rotation is stored on the same partition as
00169  * the daily rotation the copy will consist of hard linked files that
00170  * do not actually use any significant disk space.
00171  *
00172  * <li>Weekly rotations are pruned as with daily rotations. The
00173  * $this->app->localsys->rotation_total_weekly configuration parameter
00174  * defines the number of recent weekly backupsets that are maintained.
00175  *
00176  * <li>On one day of the month (specified in the
00177  * $this->app->localsys->rotation_dayofmonth configuration parameter,)
00178  * the backupset that was just created is copied to the monthly backup
00179  * rotation. This rotation can also be hardlinked to save disk space. 
00180  *
00181  * <li>Monthly rotations are pruned as with weekly rotations. The
00182  * $this->app->localsys->rotation_total_monthly configuration parameter
00183  * defines the number of recent monthly backupsets that are maintained.
00184  *
00185  * </ul>
