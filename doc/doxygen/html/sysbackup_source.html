00062  * <p><b>Original Goals:</b>
00063  *
00064  * <p>The basic goal of this script has always been to take a picture
00065  * of the server every night - in a way that makes it easy to find any
00066  * file that might be needed. In addition this picture must be
00067  * complete enough that it can be used to build a new server in the
00068  * event of a disaster or during migration. Not surprisingly it often
00069  * happens that equipment upgrades are usually scheduled to occur more
00070  * frequently than disasters!
00071  *
00072  * <p>To make this possible it must be easy for the system admin to
00073  * list all the items that need to be backed-up. The script should be
00074  * smart enough to check if the item exists on the current server and
00075  * make a backup of it - or not bother if it doesn't exist. This is
00076  * because there may be more than one server with similar but not
00077  * identical configuration. To save time the backup script must be
00078  * smart enough to live in one repo but run on the various similar
00079  * servers.
00080  * 
00081  * <p>The backups should be saved in a rotation on the current
00082  * server. The script must also be able to save a rotation on one or
00083  * more remote servers. There should be separate daily, weekly and
00084  * monthly rotations to ensure that long term backups are readily
00085  * available (long term rotations are often helpful in cases where a
00086  * problem causes damage that is not readily apparent.)
00087  *
00088  * <p>If two tar files are identical between backups the script should
00089  * be able to detect the situation and replace the second copy with a
00090  * hard link to the previous identical copy.

00187  * <p><b>FAQ</b>
00188  *
00189  * <p><b>What's the difference between a warning and a fatal error
00190  * here?</b> 
00191  *
00192  * <p>Answer: There are two answers to this question: 
00193  *
00194  * <p>(1) You will often see a few warnings that contain informat
00195  *        about an error. These are often followed by a fatal
00196  *        error. The entire message is actually a fatal error but the
00197  *        warning method is called for the information and the fatal
00198  *        method is called at the end with the final error
00199  *        message. This is a coding issue (fatal stops the program
00200  *        execution and so can only be used once at the end.)
00201  *
00202  * <p>(2) In principal the backup app must stop for configuration
00203  *        errors and sanity check errors. If there's something strange
00204  *        going on we assume that the system is either not configured
00205  *        properly or there's something seriously wrong that must be
00206  *        corrected before the backup system is allowed to run - so
00207  *        stop and let the admin fix it before trying again. Warnings
00208  *        are for problems that do not prevent the completion of the
00209  *        backup. 
00210  *
00211  *        In practice it's not always easy to decide between the two
00212  *        when writing the code. After the code has been written there
00213  *        are problems that the admin might start to want to see as
00214  *        warnings instead of fatal errors and vice-versa.

00216  * <p><b>History</b>
00217  *
00218  * <p>This file started out as a super simple bash script called
00219  * /etc/cron.daily/daily.sh on a dev server used by Sam Azer. It is
00220  * supposed to be short and straight-forward but circumstances cause
00221  * it to grow a little from time to time. It has been used on BNT
00222  * servers and servers belonging to various customers.
00223  *
00224  * <p>The code was first published in the Bell Integrator Build & Run
00225  * server maintenance manual in 2002. The ability to write nightly
00226  * backups to an off-site server was added a few years later.
00227  * 
00228  * <p>The rewrite as a PHP script was prompted by the complexity of
00229  * the daily.sh script running on mail2.bnt.ca. There are too many
00230  * operations being performed in that script (including general server
00231  * status and anti-spam reporting.) It became obvious that it would
00232  * be necessary to separate the anti-spam code from the backup code
00233  * and add some minor improvements along the way. Offsite backups, for
00234  * example, are not being rotated by daily.sh and the use of
00235  * hard-links to save disk space had been planned for years but never
00236  * implemented.
00237  *
00238  * <p>The current re-write implements a light-weight plug-in
00239  * architecture to allow the various blocks of code to be separated
00240  * from the basic backup procedure.
