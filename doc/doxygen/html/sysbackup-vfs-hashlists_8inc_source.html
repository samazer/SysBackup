00101 /**
00102  *
00103  * Hashlist class
00104  *
00105  * The hashlist code in the base IO class is getting really
00106  * complicated as I try to incorporate the code to properly handle
00107  * rotations. The solution is to factor the hashlists out of the
00108  * baseIO code so that it can be possible to have many hashlists in an
00109  * object - while still allowing the code to be readable. Here's my
00110  * first try:
00111  *
00112  * Basically a backupset consists of a set of tarballs (along with a
00113  * few compressed files; but we'll call them all tarballs.) Each
00114  * tarball has a size and a basic SHA1 hash. Later on, after
00115  * encryption, the tarball has another hash for the encrypted version
00116  * of the file but we're not dealing with that yet.
00117  *
00118  * The problem is that we don't want to add the tarball to the
00119  * backupset if there is another tarball, either in the current
00120  * backupset or in the previous backupset, that already has the same
00121  * basic SHA1 hash. To be able to do this we need to be able to find
00122  * the previous backupset for any given rotation (daily, weekly or
00123  * monthly,) and check the list of tarball hash values. There are some
00124  * complications that also need to be managed, ie: if the daily and
00125  * weekly or daily and monthly rotations are on the same partition we
00126  * need to check the hash lists for the previous backupsets in the
00127  * daily rotation instead of the previous backupset in the weekly or
00128  * monthly rotation.
00129  *
00130  * In any case, as you can see, we want to keep track of two different
00131  * hashlists as we are performing the backup: The hashlist for the
00132  * current backupset and the hashlist for the previous backupset.
00133  *
00134  * While we're doing this we will also need some stats: total number
00135  * of tarballs, total size of the backupset, total number of tarballs
00136  * that have been hardlinked and total size of the hardlinked
00137  * tarballs.
00138  *

00297    function _reverse( $hashlist )
00298    {
00299       $hashlist_rev = array();
00300       foreach ( $hashlist as $fn => $sha1 )
00301       {
00302         /**
00303          * more than one file might have the same hash but we only
00304          * care about one of them (any one) because we assume that
00305          * hardlinks will always work fine in the backupset folder
00306          * (It's the same partition for all files in that folder.)
00307          *
00308          * NOTE that this can fail miserably if rotations are saved to
00309          * a different partition and the system writes a symlink to a
00310          * previous backupset. This is an error. It's better to turn
00311          * off the linking feature completely in the case where
00312          * multiple partitions are being used.
00313          */
00314          if ( isset($hashlist_rev[$sha1]) )
00315             continue;
00316 
00317          $hashlist_rev[$sha1] = $fn;
00318       }
00319 
00320       return $hashlist_rev;
00321    }
